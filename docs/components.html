Link

Link
Markup <a href="#">Link</a>
  • Links go places, while buttons do things. The difference is that buttons change the state of the website. That can be on the server side or the client side, but a button implies that some action is being taken, and something, somewhere is different now. Links take you somewhere else, but neither the place you were nor the place you’re going have changed when you do so.
  • Yes, buttons are routinely abused to work as just bigger, more prominent links — usually along with other bad design decisions that visually mark almost all of the items on a page as the most important thing, creating frustration and increasing the user’s cognitive load. It’s a terrible practice, and just because it’s common is no reason to do it.
  • In places where you might be tempted to use a button to go somewhere because the link should be visually prominent, consider using the prominent link component instead.
  • If you need to present the user with a prominent decision point when hen could go to any of several different places, and each one calls for some significant explanation, you might consider the panel link component.

Prominent Link

Left-Aligned Link Center-Aligned Link Right-Aligned Link
Markup <a href="#" class="prominent">Left-Aligned Link</a> <a href="#" class="prominent center">Center-Aligned Link</a> <a href="#" class="prominent right">Right-Aligned Link</a>
  • Use this component when you want to give users the opportunity to go somewhere (meaning that they’re not changing the website back-end or front-end), but the link should be visually prominent.
  • This component is suitable as a call to action where the action is to go somewhere, rather than to do something. As with other calls to action, you should not have more than one on a page. The user can only do one thing; if you ask hen to do several things, you only create confusion and frustration. If you have multiple links to share in a sequence (e.g., a menu that takes up a significant part of a page’s content), consider using the panel link component.
  • Like any other option that creates visual prominence, do not overuse this component. Generally speaking, you should not use this more than once on a page. Visual hierarchy helps create clarity, unless you try to make everything prominent, then there is no hierarchy and no clarity, because everything is equally prominent. It’s the design equivalent of writing in all caps or shouting at someone.
  • If the call to action does do something (changing the state of the website, either on the front-end or the back-end), then you should use a button instead.

  • This component is best used for a menu that takes up most or all of a page (like the homepage for this design system).
  • The panel is designed to work with secondary, tertiary, and quarternary headings (<h2>, <h3> and <h4>), but not primary headings (<h1>). Each page should have only one primary heading, so if you’re considering using it in a panel, you might be using the wrong component.
  • Since the entire panel is a link, don’t use any other links or buttons inside the panel.

Text Input

Markup <input type="text" placeholder="Text input" /> <input type="email" placeholder="you@example.com" /> <input type="search" placeholder="Your search terms..." /> <input type="tel" placeholder="1 (212) 867-5309" /> <input type="url" placeholder="https://scarletbanner.com" /> <input type="number" value="0" min="0" max="10" /> <input type="text" placeholder="Text input" value="Error!" class="error" />
  • Do not use placeholders in place of labels. When you begin typing in a text box, the text disappears, so if you forget precisely what you need to enter in this box, you’ll need to remove what you’ve written. Placeholders aren’t designed to take the place of labels; that’s what labels are for. Placeholders should be used to provide an example of the sort of input a user should provide in the box.
  • The text input is the most generic of the fields provided here. This is appropriate for fields that need to collect a piece of arbitrary short text. Wherever appropriate, use one of the more specific fields detailed here. The text input is the default option that you can use when none of the other options quite fit. When asking the user for text that you expect to exceed more than a dozen or so character, use the text area component instead.
  • The email input is the field you should use when you ask the user to enter an email address. Though this may look the same on a desktop browser, the specialized field can prompt a variety of affordances. For example, on iOS Safari, the email field brings up a different keyboard, with special @ and .com buttons that make typing an email address more convenient.
  • The search input is the field you should use when you ask the user to enter a search query.
  • The telephone input is the field you should use if you ever need to ask the user to enter a telephone number.
  • The URL input is the field you should use when you ask the user to enter a URL.
  • The number input is the field you should use when you ask the user to enter a number.
  • Never ask for more data than you actually need. Every additional field on a form increases the interaction cost, reducing the likelihood that you’ll get any response at all. Interrogate every field to determine if it’s something you really need, and eliminate everything that is not absolutely necessary in order to provide the service that you wish to provide.

Text Area

Markup <textarea class="sm">This is a small text area.</textarea> <textarea>This is the standard text area.</textarea> <textarea class="lg">This is a large text area.</textarea>
  • The text area input is appropriate when you’re asking a user to enter a significant amount of text (anything more than a dozen or so characters). For short bits of text, you can use a text input field.
  • Just like other form elements, text areas should appear with a concise and informative label.

Dropdown

Markup <select> <optgroup label="Chromatic Dragons"> <option value="red">Red Dragon</option> <option value="blue">Blue Dragon</option> <option value="green">Green Dragon</option> <option value="black">Black Dragon</option> <option value="white">White Dragon</option> </optgroup> <optgroup label="Metallic Dragons"> <option value="gold">Gold Dragon</option> <option value="silver">Silver Dragon</option> <option value="bronze">Bronze Dragon</option> <option value="brass">Brass Dragon</option> <option value="copper">Copper Dragon</option> </optgroup> </select>
  • Use a dropdown when you would like a user to choose a single option from a set list of options. When to use a dropdown and when to use a set of radio buttons can be subtle, but some reasons to prefer a dropdown over radio buttons include:
    • When there is a clear default, recommended, or most common option (which you should have selected by default).
    • There are quite a few options to choose from, but it’s easy for users to notice the pattern and predict what the available options will be (e.g., a list of countries).
    • When you have more than seven options to choose between.

Radio Buttons

Markup <label class="radio"> <input type="radio" name="radio-example" value="yes" id="radio-example-yes" /> Yes </label> <label class="radio"> <input type="radio" name="radio-example" value="no" id="radio-example-no" /> No </label> <label class="radio"> <input type="radio" name="radio-example" value="note" id="radio-example-note" /> Well… <p>This radio button’s label has a note.</p> </label> <label class="radio extended"> <input type="radio" name="radio-example" value="ext" id="radio-example-ext" /> You may find on some occasions that it is useful... </label>
  • Use radio buttons when you would like a user to choose a single option from a set list of options. When to use radio buttons and when to use a dropdown can be subtle, but some reasons to prefer radio buttons over a dropdown include:
    • When there is no clear default, recommended, or most common option. When this is the case, do not select any option by default (though you may want to require that the user select an option on the back-end).
    • When it’s important that users be able to compare the options side-by-side.
    • When there are five or fewer options to choose from.

Checkboxes

Markup <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="fighter" id="checkbox-example-fighter" /> Fighting Person </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="wizard" id="checkbox-example-wizard" /> Magic User </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="thief" id="checkbox-example-thief" /> Thief </label> <label class="checkbox"> <input type="checkbox" name="checkbox-example" value="cleric" id="checkbox-example-cleric" /> Cleric </label> <label class="checkbox extended"> <input type="checkbox" name="checkbox-example" value="extended" id="checkbox-example-extended" /> Actually... </label>
  • Use a set of checkboxes when you would like a user to choose one or more options from a set list of options.
  • You can also use a single checkbox on its own to capture a boolean value from a user, like opting in to an agreement.
  • If you use a checkbox to gather consent or agreement from a user, it cannot be checked by default. Consent is only ethical or binding when it is freely and consciously given. It is entirely too easy for a user to submit a form without noticing a checkbox that is marked by default. Besides being a deeply unethical dark pattern or ”antipattern,“ in many legal jurisdictions this will not stand up as legal consent.

Figure

Markup <figure> <img src="/path/to/image.jpg" alt="Always provide alt text" /> <figcaption>Caption text</figcaption> </figure>
  • Though perhaps most commonly used with images, figures can also feature videos, tables, or other media.
  • You don’t have to add Figure X. to your figure captions. That is added automatically by CSS. You can exempt a figure from this counting by applying the uncounted class to the <figcaption> element (e.g., <figcaption class="uncounted">). If there is a Figure 1 before an uncounted figure, then the next figure after it will be Figure 2. The uncounted figcaption will display its own content and nothing else.

  • Tables are for displaying tabular data. Though a less common sin against semantic markup these days, it’s still important to only use tables to present tabular data, and not for presentation or layout. This is by no means a general injunction to avoid using tables; there’s a lot of tabular data in the world, and you absolutely should use tables to present it. As always, good semantic markup is all about considering the content you’re presenting and what it means first and foremost.
  • You don’t need to number your tables manually; the CSS will add numbering automatically. This also means that if you rearrange the tables on a page, or if your content is reused on another page, the numbering will still be correct. You can exempt a table from this counting by applying the uncounted class to the <caption> element (e.g., <caption class="uncounted">). If there is a Table 1 before an uncounted table, then the next figure after it will be Table 2. The uncounted caption will display its own content and nothing else.
